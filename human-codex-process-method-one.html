Method One / Team Introduction

Human-Codex Process

A controlled way to use AI agents for product work: not by bypassing professional practice, but by making product intent, role authority, documentation, gates, and evidence executable.

Talk track: This is not a demo of a clever prompt. It is a proposed operating model for letting agents do real work while keeping human expertise and control points visible.

Context

Vibe coding needs a quality system.

AI can produce fast output, but consistency breaks when every run depends on one chat's memory, implied intent, and whatever the model decides to do next.

What breaks

  • Output quality varies wildly between sessions and agents.
  • Product memory lives in one conversation and does not transfer cleanly.
  • Agents overbuild, spin on empty, give up, or drift down the wrong path.

What fixes it

  • Product brief, UX, architecture, and operations docs as shared memory.
  • Gates and evidence so humans steer without micromanaging every edit.
  • Role separation so product, build, QA, release, and ops do not blur.
Framing: AI automates software development. It does not make sound development practices obsolete.
Reality Check

This does not replace knowing how software is built.

The surprise is not that AI needs briefs, architecture, tests, QA, and release gates. The surprise is that it can now produce and execute much of that work for us.

The wrong expectation

  • AI lowers workload by skipping process.
  • Anyone can now build reliable software without knowing the craft.
  • If the model fails, it proves AI is only junior-developer level.

The useful expectation

  • AI lowers workload by doing the professional steps with us.
  • Human expertise still decides what good looks like.
  • The process turns expert expectations into reusable agent instructions.
Warning: This helps professionals move faster and produce more consistent software. It does not turn non-professionals into software professionals.
Model

Separate authority, intent, technique, and visual identity.

The process makes the context explicit, so an agent knows what it is allowed to decide and what it must hand off.

Key shift: The agent is not one unbounded assistant. It behaves as a role with a lane, inputs, outputs, and stop conditions.
Lifecycle

A delivery chain with evidence at the gates.

The lifecycle can be lightweight, but the handoffs are explicit: product intent, design, architecture, ready issues, TDD evidence, QA evidence, release evidence, and operations ownership.

Coordination

Parallel work is useful only when the return point is controlled.

The Product Owner can coordinate subprocesses in parallel, but the human still sees the control thread: questions, documents ready for approval, status, evidence, and deployment gates.

What runs in parallel

  • UX and Architecture after brief approval.
  • Technical Lead planning and Operations planning.
  • Developer work and later QA/release validation in separate workspaces.
Guardrail: parallel channels do not erase role authority. They return artifacts or questions to the controlling conversation.
Usage

Install once, then invoke the process as a skill.

The skill is a thin wrapper around the live repository. It does not copy the process; it points Codex at the checkout and re-reads the current files.

Create the skill

Create a Codex skill named `process` that I can invoke with `$process`. Make it a thin wrapper around this checkout: do not copy repo content, and re-read the live files on each use. Support role prompts like `role=PO`, `role=Developer`, `role=QA`, and show me the created `SKILL.md`.

Use the skill

$process role=PO Interview me and create a product brief for <product>.

From there, you can either call roles directly or have the Product Owner coordinate subprocesses.

Practical benefit: updating the process is a normal repository update, not an admin chore.

Discussion

Where should we use this first?

How should or can we use this in our work on Method One?

Good first candidates

  • Small features with clear acceptance criteria.
  • Internal tools or workflow automation.
  • Defect streams where TDD + QA evidence would pay off.

Team choices to make

  • Which gates are mandatory for Method One?
  • What artifact budget fits our normal work?
  • Where do we want humans to approve, and where can agents proceed?
Close: We do not need to adopt everything at once. Pick one real Method One slice and test the workflow end to end.
