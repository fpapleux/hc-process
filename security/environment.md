# Environment and Secrets

This document defines the target direction for environment and secret handling.

## Operator Vault

`environment.local` is the operator's personal vault. It must not be treated as
a general agent configuration file.

Target direction:

- Move the operator vault out of the OpenClaw working directory.
- Store it under an operator-owned private path.
- Restrict permissions to the operator account.
- Do not mount or expose it to ordinary agents.
- Do not let product scripts assume it exists.

Recommended future shape:

```text
operator vault:
  ~/.config/openclaw/private/environment.local

role/product scoped runtime config:
  generated by approved tooling
  contains only variables required for that role/product/session
```

The exact path can change, but the boundary must not: the operator vault is not
an agent-readable shared file.

## Role-Scoped Configuration

Agents receive configuration through scoped mechanisms:

- role-scoped environment file;
- product-scoped environment file;
- approved MCP server;
- approved secret provider;
- explicit non-secret repo configuration.

Each scoped config contains only the values required for that agent's assigned
role, product, and technology.

## Rules

- Do not hard-code secrets.
- Do not commit secrets.
- Do not paste secrets into GitHub Issues, PRs, release notes, logs, or test
  evidence.
- Do not read the operator vault from agent code.
- Do not pass production credentials to developer or QA agents.
- Fail closed when required configuration is missing.

## Environment Promotion

Products with mutable external systems use explicit environments:

```text
local/dev -> test/staging -> production
```

Scripts do not use casual `--prod` switches.

Instead:

- Developer scripts target local/dev resources.
- QA scripts target test/staging resources.
- Release/operations tooling targets production only after promotion gates pass.
- Target environment is supplied by the role/product process and scoped
  credentials, not by ad hoc command-line choice.
